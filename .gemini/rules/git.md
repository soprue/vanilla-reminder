# Git Workflow & Commit Rules

이 문서는 프로젝트의 변경 사항을 논리적으로 분리하여 커밋하고 관리하는 절차를 정의합니다.

## 1. 커밋 원칙 (Commit Principles)
- **논리적 단위 커밋**: 하나의 커밋은 하나의 의미 있는 변경(기능 추가, 버그 수정 등)만 담아야 합니다.
- **명시적 파일 추가**: `git add .` 또는 `git add -A` 사용을 금지합니다. 수정된 파일을 개별적으로 명시하여 추가하십시오.
  - 예: `git add src/core/Component.ts` (O)
  - 예: `git add .` (X)
- **사용자 승인 필수**: 커밋을 실행하기 전, 반드시 사용자에게 **커밋 계획(파일 목록 및 메시지)**을 제시하고 승인을 받아야 합니다.

## 2. 커밋 메시지 컨벤션 (Conventional Commits)
메시지는 영문 작성을 기본으로 하며, 아래 형식을 따릅니다.
- **형식**: `<type>: <description>`
- **타입(Type)**:
  - `feat`: 새로운 기능 추가
  - `fix`: 버그 수정
  - `docs`: 문서 수정 (GEMINI.md, README 등)
  - `style`: 코드 포맷팅, 세미콜론 누락 등 (코드 변경 없음)
  - `refactor`: 코드 리팩토링
  - `test`: 테스트 코드 추가/수정
  - `chore`: 빌드 업무, 패키지 매니저 설정 등
- **제목**: 50자 이내로 간결하게 작성합니다.

## 3. 워크플로우 단계 (Workflow Steps)
1. **변경 확인**: `git status`와 `git diff`를 통해 변경된 내용을 파악합니다.
2. **논리적 분류**: 변경 사항을 위 `type`에 따라 그룹화합니다.
3. **계획 제시**: 사용자에게 각 그룹별 커밋 메시지와 포함될 파일 목록을 제안합니다.
4. **승인 및 실행**: 사용자가 승인하면 순차적으로 커밋을 실행합니다.
5. **확인**: `git log --oneline -n 5`로 커밋 결과를 보고합니다.

---
*주의: 민감한 정보(.env, secrets 등)가 포함되지 않도록 항상 주의하십시오.*
